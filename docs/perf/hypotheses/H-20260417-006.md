# Hipótese H-20260417-006 — staleTimes.dynamic client router cache

## Believe That
Configurar `experimental.staleTimes.dynamic: 30` em [next.config.ts](../../../next.config.ts).

**Sintoma reportado pelo user** (navegação mobile real):
- 1ª vez home → `/busca`: RÁPIDO (prefetch client)
- Volta home, 2ª+ vezes → `/busca`: LENTO (cache invalidado)
- Swipe-to-refresh (F5): RÁPIDO de novo (caminho diferente, full reload)

## Will Result In

- **Navegação home ↔ /busca instantânea por 30s após visita** (cache client reutilizado)
- **Renderização qualitativamente percebida como rápida** na 2ª+ navegação
- Sem impacto em Lighthouse synthetic (Lighthouse roda cold start, não testa navegação)
- Sem impacto em bundle, TBT, LCP, CLS

**Mecanismo (verificado em Next types):**

```typescript
// node_modules/next/dist/server/config-shared.d.ts
staleTimes?: {
  dynamic?: number;  // default 0 em Next 16 → cache client expira imediato
  static?: number;   // min 30s
};
```

Rotas dinâmicas (`ƒ` no build — `/busca`, `/lancamentos`, `/apartamentos-curitiba`, `/casas-curitiba`, etc) têm **cache client-side zero** por padrão. Cada navegação dispara nova request ao server via RSC payload, independente do prefetch ter acontecido.

Setar `dynamic: 30` instrui o Next: *"por 30s após buscar essa rota, reuse o cache em navegações subsequentes sem re-fetch ao server"*.

## Success Criterion

**Qualitativo (validado pelo user em mobile real):**
- Fluxo home → /busca → home → /busca → home → /busca dentro de 30s deve sentir-se **instantâneo** na 2ª e 3ª navegações
- Sintoma reportado desaparece

**Quantitativo (não aplicável em Lighthouse synthetic):**
- Bundle /busca: inalterado (flag de config, zero KB)
- Build passa sem warnings
- Todas as outras métricas: iguais (Lighthouse mede cold start)

## Kill Criterion

Reverter se:
- Build quebrar
- Dados extremamente stale aparecerem (ex: imóvel vendido reportado por user continuando visível por > 1min)
- Alguma feature dependente de fresh state quebrar (favoritos, filtros aplicados)
- User reportar UX pior que antes (confirmação subjetiva)

## Baseline Measurement

Baseline do sintoma é **qualitativo**:
- Reportado pelo user em 2026-04-17: "primeira vez rápido, segunda em diante demora muito mais, só F5 resolve"
- Reprodutível manualmente no iPhone/Android navegando pela BottomNav

**Quantitativo (não aplicável):** Lighthouse sempre mede cold start, então não captura degradação de navegação quente.

## Predicted Delta

- **Navegação quente**: instantânea (cache hit)
- **Bundle**: 0 KB
- **Build time**: 0s
- **Dados stale**: até 30s após primeira visita
- **Impacto em Lighthouse**: zero (teste cold start)

## Attribution Mechanism

### Pós-deploy (validação)
1. Abrir `/busca` em mobile
2. Navegar `/` (home)
3. Clicar "Buscar" no BottomNav → deve ir **instantaneamente** (usando cache de 30s)
4. Voltar `/`, aguardar > 35s, clicar "Buscar" → deve fazer re-fetch (cache expirou)
5. Validação passa se navegação dentro de 30s é imediata e fora de 30s comporta normalmente

**Teste de staleness:**
- Setar cache hit
- Alterar algo no CRM Loft (mock via admin — ou só simular)
- Navegar de volta antes de 30s → deve mostrar estado antigo (esperado)
- Navegar depois de 30s → deve mostrar novo estado
