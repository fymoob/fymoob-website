# Hipótese H-20260417-006 — staleTimes.dynamic client router cache

## Believe That
Configurar `experimental.staleTimes.dynamic: 30` em [next.config.ts](../../../next.config.ts).

**Sintoma reportado pelo user** (navegação mobile real):
- 1ª vez home → `/busca`: RÁPIDO (prefetch client)
- Volta home, 2ª+ vezes → `/busca`: LENTO (cache invalidado)
- Swipe-to-refresh (F5): RÁPIDO de novo (caminho diferente, full reload)

## Will Result In

- **Navegação home ↔ /busca instantânea por 30s após visita** (cache client reutilizado)
- **Renderização qualitativamente percebida como rápida** na 2ª+ navegação
- Sem impacto em Lighthouse synthetic (Lighthouse roda cold start, não testa navegação)
- Sem impacto em bundle, TBT, LCP, CLS

**Mecanismo (verificado em Next types):**

```typescript
// node_modules/next/dist/server/config-shared.d.ts
staleTimes?: {
  dynamic?: number;  // default 0 em Next 16 → cache client expira imediato
  static?: number;   // min 30s
};
```

Rotas dinâmicas (`ƒ` no build — `/busca`, `/lancamentos`, `/apartamentos-curitiba`, `/casas-curitiba`, etc) têm **cache client-side zero** por padrão. Cada navegação dispara nova request ao server via RSC payload, independente do prefetch ter acontecido.

Setar `dynamic: 30` instrui o Next: *"por 30s após buscar essa rota, reuse o cache em navegações subsequentes sem re-fetch ao server"*.

## Success Criterion

**Qualitativo (validado pelo user em mobile real):**
- Fluxo home → /busca → home → /busca → home → /busca dentro de 30s deve sentir-se **instantâneo** na 2ª e 3ª navegações

**Quantitativo (não aplicável em Lighthouse synthetic):**
- Bundle /busca: inalterado (flag de config, zero KB)

**SUCCESS** — confirmada empiricamente pelo user. Mudança mantida em prod.

### Por que é um SUCCESS limpo

1. **Predição casou com observação**: predicted "navegação instantânea por 30s após visita", user confirmou "resolveu, era exatamente isso"
2. **Zero regressão**: mudança é config client-router apenas, zero impacto em bundle, TBT, LCP, CLS synthetic
3. **Trade-off aceitável**: 30s staleness tolerável para inventário imobiliário
4. **Validação qualitativa apropriada**: não precisa Lighthouse (synthetic não captura navegação quente)

### Decisão sobre Nível 2 (prefetch manual via `router.prefetch`)

Avaliado e **não aplicado** — over-engineering. Next `<Link>` já faz prefetch automático via IntersectionObserver. Combinado com `staleTimes: 30`:

1. User entra home → BottomNav visível → `<Link href="/busca">` prefetch automático
2. Cache client TTL 30s → cache hit na navegação
3. Volta home → Link re-prefetcha (IntersectionObserver trigger)
4. Próxima navegação → cache fresh

Adicionar `router.prefetch` manual seria:
- +30-50KB request RSC em background a cada home mount
- Zero ganho perceptível (comportamento já é aproximadamente ideal)
- Mais código/superfície pra bug

Se em 1-2 semanas aparecer caso específico ainda lento, revisitar com H-007.

### Lição

Config `staleTimes.dynamic: 30` é **a solução correta** pra esse sintoma em Next 16. Não é hack — é ajuste do default agressivo (0s) que a Vercel escolheu priorizando freshness sobre navegação. Para sites com inventário de atualização lenta (imobiliário, e-commerce de nicho, blog), o trade-off inverte.

Mantida em prod. Sessão de navegação quente fechada.
