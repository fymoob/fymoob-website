# Hipótese H-20260417-005 — SortDropdown mobile: Sheet → `<select>` nativo

## Believe That
Substituir `<Sheet>` do `@base-ui/react` por `<select>` HTML nativo no mobile de [src/components/search/SortDropdown.tsx](../../../src/components/search/SortDropdown.tsx).

Atualmente (confirmado por leitura do arquivo):
- **Desktop**: já usa `<select>` nativo ✅
- **Mobile**: usa `<Sheet>` + `useState<open>` + lista de `<button>` custom

## Will Result In

- **UX mobile melhora** — iOS abre o wheel picker nativo (otimizado para toque)
- **Client component ligeiramente mais leve** — remove import `Sheet/SheetContent/SheetTrigger/SheetTitle` + useState
- **Bundle `/busca`:** `MEASUREMENT_REQUIRED` — Sheet ainda é usado em SearchBar mobile + Header, então chunk de Sheet continua carregando. Ganho esperado pequeno (<5KB) ou zero.
- **TBT:** ganho marginal (~5-15ms) por menos hidratação nesse componente
- **FCP/LCP:** neutro

**Atenção à predição realista:** diferente do que Agent 3 estimou como "maior ganho e menor risco" do Tier 2, a evidência de uso compartilhado do Sheet (SearchBar, Header) sugere ganho de bundle próximo de zero. O principal ganho é **UX**, não perf.

## Success Criterion

Apenas dois critérios hard (não há ganho significativo esperado):
- **Build passa** sem erros
- **Visual manual**: `<select>` no mobile iOS abre picker nativo, desktop continua idêntico
- TBT não regredir > 20ms
- Bundle não crescer > 5KB

Sem success criterion numérico agressivo — é uma mudança de qualidade de código + UX, não perf claim.

## Kill Criterion

Reverter se:
- Build quebrar
- Picker mobile não abrir ou style horrível em Android Chrome
- TBT regredir > 50ms (muito improvável, mas possível)
- Bundle crescer inexplicavelmente > 20KB
- Hidratação visual quebrar (layout shift novo)
