# Hipótese H-20260417-004 — experimental.inlineCss

## Believe That
Habilitar `experimental.inlineCss: true` em [next.config.ts](../../../next.config.ts). Next 16.1 + Turbopack substitui `<link rel="stylesheet">` por `<style>` inline no HTML, eliminando request render-blocking de CSS.

## Will Result In

- **LCP mediana cai 100-300ms** — CSS chega junto com HTML, eliminando round-trip de download + parse sequencial
- **FCP mediana cai 150-250ms** — first paint não bloqueado por CSS pendente
- **TBT neutro ou marginal** — não muda JS execution
- **CLS possivelmente melhora** — CSS chega imediatamente, evita paint sem styles que depois shifta
- **HTML cresce ~25KB** — CSS principal `f75161fcad82fcb2.css` (25KB) será inline

**Mecanismo:** Lighthouse v5 audit `render-blocking-resources` apontou `f75161fcad82fcb2.css` (25 KB) como bloqueando FCP ~200ms e LCP potencialmente. Com `inlineCss: true`, esse CSS fica embutido no HTML servido pelo Vercel Edge → zero request adicional, zero round-trip, CSS disponível no parse do HTML.

**Trade-off:** HTML maior (não cacheável entre page loads). Mas gain em First Paint > cost em bytes de network (CSS é entregue uma vez vs múltiplas requests sem cache hit).

## Success Criterion

Em `/busca` mobile, 5 runs pós-deploy:
- **LCP mediana: < 4.70s** (baseline atual pós-H003: 4.89s; meta -200ms = -4%+)
- **FCP mediana: < 1.10s** (baseline 1.23s; meta -100ms)

## Kill Criterion

Reverter `inlineCss: true` → `false` se:
- LCP piorar > 100ms (mediana pós > 4.99s)
- FCP piorar > 80ms
- HTML size crescer > 40KB (overhead demais)
- TBT regredir > 50ms
- Qualquer erro de CSS ausente (componentes renderizando sem estilo)

## Conclusão

**FAILED — KILL TRIGGERED — mudança REVERTIDA no commit seguinte.**

### O que deu errado

1. **Predicted HTML +20-30KB virou +326KB real.** Muito acima da previsão. CSS total (Tailwind + componentes + gradient backgrounds + tree-shaken do @base-ui etc) é muito maior que os "25KB render-blocking" reportados pelo Lighthouse render-blocking-resources audit. Esse audit reporta só o CSS inicial crítico, não o CSS total da página.

2. **HTML +326KB → main thread parse 3x mais HTML** → TBT subiu +237ms (parse + style calculation).

3. **HTML maior → tempo até HTML estar todo carregado cresceu** → LCP subiu +238ms. Inline CSS acelerou FCP (-155ms) mas atrasou o paint final do LCP element.

4. **CLS piorou** inesperadamente — provavelmente porque inline CSS aplica em ordem diferente do layout final, causando mais shifts.

### Lições aprendidas

**Meta-lição:** benchmark externo de "-100-300ms LCP" do `inlineCss` assume CSS **pequeno e crítico**. Para páginas com CSS grande (Tailwind + muitos componentes), o trade-off inverte.

**Lição sobre render-blocking-resources audit:** esse audit diz quanto do CSS atual é blocking para FCP, não quanto de CSS vai virar HTML bytes se inlinado. Números diferentes.

**Lição sobre atribuição:** `curl | wc -c` pré-deploy teria revelado +326KB real vs +25KB esperado. Devia ter rodado essa verificação no ambiente local (`npm run build && npm start`) antes de commitar. Adicionar isso ao protocolo para próxima hipótese que toque em HTML size.

### Decisão

- **Reverter** `inlineCss: true` (commit seguinte)
- **Não tentar `inlineCss` em /busca** nem outras páginas com Tailwind
- Pode ser útil em páginas MUITO simples com pouco CSS (ex: `/politica-de-privacidade`) — fora de escopo
