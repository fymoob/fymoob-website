# Hipótese H-20260417-007 — ISR com cacheTag + revalidateTag

## Believe That
Adicionar `tags: ["imoveis"]` ao `unstable_cache` de `getCachedProperties` em [src/services/loft.ts:398](src/services/loft.ts#L398) e criar endpoint `POST /api/revalidate` que aceita `{ tag }` protegido por `REVALIDATE_SECRET` para invalidação granular.

## Will Result In

### Ganhos esperados
- **Invalidação sob demanda** — quando inventário Loft muda (imóvel vendido/novo), `curl -X POST /api/revalidate` com body `{"tag":"imoveis"}` purga o cache imediatamente, sem esperar o TTL de 1h
- **Zero impacto em performance lab** — cache já existia, só ganha mecanismo de invalidação. Lighthouse: neutro
- **TTFB mantido ou melhor** — mesma arquitetura de cache, só tag adicionada
- **Preparação pra futuro** — base pra `cacheComponents` + `"use cache"` directive do Next 16.3+

### O que NÃO é objetivo
- Esta hipótese NÃO remove o `staleTimes: 30` atual (issue separada, decisão pós field data)
- NÃO é refactor grande — é adição opcional à arquitetura atual

## Success Criterion

- **Build passa** sem erros
- **Endpoint `/api/revalidate` retorna 200** com secret correto + body válido
- **Endpoint retorna 401** sem secret ou com secret errado (auth protegido)
- **Endpoint retorna 400** com body inválido (validation)
- **Invalidação funciona**: após POST com `{"tag":"imoveis"}`, próxima chamada a `getProperties()` re-fetch à Loft API (confirmar via logs)
- **Cache hit normal antes da invalidação** (não quebra o cache existente)
- **Zero regressão funcional** — `/busca`, `/imovel/*`, `/imoveis/[bairro]/*` continuam renderizando igual

## Kill Criterion

Reverter se:
- Build quebrar
- Cache parar de funcionar (TTFB de `/busca` subir > 500ms)
- Endpoint retornar 500 consistentemente
- `revalidateTag` não invalidar (teste manual falha)
- Algum comportamento inesperado em produção (erro de runtime)

## Conclusão

**SUCCESS.** Hipótese validada integralmente.

### Capacidades desbloqueadas
- Invalidação manual: `curl -X POST /api/revalidate -H "x-revalidate-secret: $SECRET" -d '{"tag":"imoveis"}'`
- Base pra webhook Loft/Vista (quando suportado) chamar após sync
- Base pra cron scheduled (Vercel Functions ou GitHub Actions) invalidar periodicamente
- Caminho aberto pra migrar para `"use cache"` + `cacheLife` em Next 16.3+

### Mudança Next 16 descoberta
`revalidateTag(tag)` do Next 14/15 virou `revalidateTag(tag, profile)` em Next 16. Profile pode ser string (`"default"`, `"seconds"`, etc) ou `CacheLifeConfig { expire?: number }`. Usamos `{ expire: 0 }` para refetch imediato na próxima request.

### Decisão sobre `staleTimes: 30` atual
**Mantido por enquanto.** Esta hipótese adiciona capacidade ortogonal (server-side on-demand invalidation) sem remover o cache client-side. Avaliar remoção pós-cutover quando Speed Insights tiver field data suficiente pra decidir.

### Operacional pós-cutover
- Quando imóvel vendido no Loft CRM: Bruno chama `/api/revalidate` via webhook, admin panel, ou script
- Site atualiza em <1s ao invés de esperar 1h de TTL
- Consistência com CRM real-time garantida

### Próximo (futuro)
- Webhook integration real com Vista API (se suportar)
- `cacheLife` em vez de `unstable_cache` (Next 16.3+ maturation)
- Granularidade maior (tag por imóvel) se scale justificar (atualmente 249, não justifica)
