# Hipótese H-20260417-002 — Investigação do CLS remanescente

Follow-up de [H-20260417-001](H-20260417-001.md) (PARTIAL): mesmo após `font-display: optional`, o CLS ainda aparece em ~30% das amostras com valor exato **0.0778** (determinístico).

## Forense realizada

### Attribution via `layout-shifts` audit

Extraído de 5 raw JSONs do Lighthouse onde CLS > 0:

```
Element:  <main class="min-h-screen overflow-x-clip">
Selector: body.antialiased > main.min-h-screen
Score:    0.077764 (sempre exato)
```

O shift afeta o `<main>` **inteiro** (altura 10571px × width 412px). Não é um elemento específico — é todo o conteúdo abaixo do `<Header>` fixo deslocando.

### Network timing (shift vs no-shift)

Comparando runs com e sem shift:

| Métrica | Run SHIFT (v1 run 1) | Run NO-SHIFT (v1 run 2) |
|---------|---------------------|-------------------------|
| FCP | 1239ms | 1218ms |
| LCP | 4839ms | 5268ms |
| TBT | 514ms | 527ms |
| Recursos | Mesmos (HTML, Satoshi, CSS, JS, primeira Image) | Mesmos |

**Descoberta:** timing absoluto de requests é `null` no report (Lighthouse 13 com simulated throttling) — não é possível comparar exatamente quando a font Satoshi chegou.

### Hipótese de causa (não confirmada por medição)

A `Satoshi-Variable.woff2` tem métricas (ascent/descent/x-height) diferentes da fallback Arial que o Next injeta automaticamente. Quando a font chega **antes** do block period de `display: optional` (~100ms), ela é aplicada e causa shift relativo ao fallback. Quando chega **depois**, o browser mantém fallback permanentemente → sem shift.

**Evidência circunstancial:**
- Shift é determinístico (0.0778 exato em todos os runs com CLS > 0)
- Shift rate caiu de 40% (swap) → 30% (optional) mas não zerou
- 30% sugere timing jitter no block period, não variance medidora

**Não foi confirmado via medição** porque Lighthouse não expõe timestamps individuais no report simulated. Para confirmar, seria necessário:
- Capturar via PerformanceObserver em RUM (`@vercel/speed-insights` já instalado — aguardando 28d de coleta)
- Ou instrumentar `chrome --trace-startup` manualmente

## Opções de fix consideradas

### Opção 1 — Adicionar `adjustFontFallback` explícito
Next/font já faz isso por padrão (`'Arial'`). Configurar mais finamente exige computar `size-adjust`, `ascent-override`, etc. — requer `fontkit` pra ler métricas reais da woff2. **Effort alto, ganho `MEASUREMENT_REQUIRED`**.

### Opção 2 — `display: block` (regride UX)
Bloqueia render até font chegar ou 3s timeout. Zero shift por font, mas FOIT visível em 3G. **Regressão de UX**, descartado.

### Opção 3 — Trocar Satoshi por Inter (next/font/google)
Google Fonts vêm com overrides calibrados pelo Next. Mudança grande de design, precisa aprovação do cliente. **Fora de escopo técnico.**

### Opção 4 — Altura explícita em containers críticos
Dar `min-h-[N]px` em wrappers conhecidos. Mas shift é do `<main>` inteiro — não há 1 elemento para corrigir.

### Opção 5 — Aceitar trade-off documentado (**escolhida**)
- Mediana de CLS em 5 runs lab = **0** (passa CWV threshold 0.1)
- p75 em synthetic = ~0.078 (borderline, mas field data costuma ser melhor)
- Ganho marginal de corrigir não vale o effort
- **Aguardar Speed Insights** (28d) pra validar field data real antes de investir mais esforço aqui

## Decisão

**STOP.** Não vou tentar fix cirúrgico agora. Racional:

1. **Attribution incompleta** — não consigo provar qual recurso especificamente shifta o main (timing ausente no report)
2. **Ganho marginal potencial** — CLS mediana já é 0. Corrigir 30% intermitência não move CWV agregado
3. **Esforço desproporcional** — opções 1-4 são todas ou risco alto ou MEASUREMENT_REQUIRED
4. **Dados reais virão em breve** — Speed Insights (instalado no commit `e935687`) vai dar p75 real de field data em ~28d. Se field data mostrar CLS p75 < 0.1, problema é synthetic noise. Se > 0.1, aí sim vale investigar com dados reais
5. **ROI melhor em outras hipóteses** — Tier 1.B (React Compiler) ou 1.D (inlineCss) têm ganhos previstos maiores e evidência externa sólida
